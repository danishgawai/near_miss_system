# Near-Miss Incident Detection System (CPU-Friendly)

A practical computer-vision pipeline for traffic safety analysis that detects
and classifies near-miss events from standard CCTV footage — no GPU required.

**Core capabilities**

- Object detection + multi-object tracking (YOLO + ByteTrack)
- Bird's-Eye View (BEV) projection via homography calibration
- 11-gate near-miss evaluation pipeline (age, border, proximity, heading,
  lateral separation, closing speed, CPA-TTC, persistence, and more)
- Explicit Risk Index: `RI = α·R_d + β·R_ttc + γ·R_v`
- Edge-to-edge BEV distance (not centre-to-centre)
- Per-class-pair TTC thresholds (car–car, car–pedestrian, etc.)
- Single-object events: hard braking, sudden swerve (multi-frame confirmed)
- Risk levels: High / Medium / Low
- Video annotations + JSON report + HTML dashboard + CSV telemetry log

Also included:
- `run_tracker.py` — quick detector/tracker smoke test
- `bev_calibrator.py` — interactive homography + scale calibration

## Step 2 — BEV Calibration

The near-miss engine works in metric BEV space (metres), so accurate homography
calibration is essential. Run this once per camera viewpoint.

### Run

```bash
python bev_calibrator.py \
    --video your_video.mp4 \
    --frame-index 100 \
    --output-config bev_config.json
```

### Calibration UI

The tool opens a side-by-side window:

- **Left panel** — real camera frame (source points)
- **Right panel** — BEV grid canvas (destination points)

Click at least 4 matched point pairs (alternate between left and right panels).

| Key | Action |
|---|---|
| Left click | Add a point (alternates left → right automatically) |
| `r` | Reset all points |
| `Enter` | Compute homography (requires ≥ 4 matched pairs) |
| `s` (preview) | Save homography and proceed to scale calibration |
| `q` (preview) | Quit without saving |

### Point selection best practices

- Use **4–8 well-spread pairs** covering the full drivable area
- Avoid collinear point sets
- Prefer stable ground-plane landmarks: lane corners, stop-line intersections,
  road markings with known geometry
- Avoid points near the frame edges — homography accuracy degrades there

### Scale calibration

After the homography preview appears:

1. Click 2 points on the preview that correspond to a known real-world distance
2. Press `Enter`
3. Type the real-world distance in metres (e.g. lane width `3.5`)

The tool computes `pixels_per_meter` and writes it to the config.

### Output

`bev_config.json` containing:

```json
{
  "homography_matrix": [[...], [...], [...]],
  "pixels_per_meter": 42.5,
  "calibration_points": { ... }
}
```

## Near-Miss Logic Summary

The engine evaluates every pair of tracked objects through 11 ordered gates
before computing a risk score. A pair must pass **all** gates on each frame,
and must do so for `incident_persist_frames` consecutive frames, before an
incident is fired.

### Gate order

| # | Gate | Rejects |
|---|---|---|
| 1 | Cooldown | Pairs that recently fired an incident |
| 2 | Track age | Tracks too young — velocity EMA not yet stable |
| 3 | Frame border | Tracks near the frame edge — unreliable BEV homography |
| 4 | BEV positions | Tracks without enough history for velocity |
| 5 | Proximity | Pairs farther apart than `proximity_gate_m` |
| 6 | Minimum speed | Both tracks nearly stopped — heading undefined |
| 7 | Heading validity | Either track too slow for a reliable heading estimate |
| 8 | Scenario + lateral | Wrong lateral geometry for the classified scenario |
| 9 | Closing speed | Diverging pairs — moving apart, not towards |
| 10 | Track quality | Low confidence or direction consistency |
| 11 | CPA miss distance | Projected trajectories will not come close enough |
| 12 | Persistence | Condition not sustained for N consecutive frames |

### Risk scoring

Pairs that pass all gates are scored with the **Near-Miss Risk Index**:

```
RI = α·R_d  +  β·R_ttc  +  γ·R_v

R_d   = 1 − edge_dist / ri_distance_ref     (spatial proximity)
R_ttc = 1 − ttc / ri_ttc_ref                (temporal urgency)
R_v   = speed_diff / ri_v_max               (motion intensity)
```

**Key design decisions:**

- **Edge-to-edge distance** is used for scoring (not centre-to-centre).
  Each vehicle's bbox footprint is subtracted before computing R_d, so large
  vehicles are not penalised for their physical size.
- **Class-pair TTC thresholds** (car–car, car–pedestrian, etc.) act as binary
  gates — a pair is rejected if its TTC exceeds the threshold. They are
  **not** used as the scoring denominator.
- `ri_distance_ref` and `ri_ttc_ref` are separate configurable references
  used only for normalising R_d and R_ttc within the scoring formula.

**Default risk thresholds:**

| RI value | Risk level |
|---|---|
| ≥ 0.65 | **High** |
| ≥ 0.40 | **Medium** |
| < 0.40 | **Low** |

Vulnerable-vs-heavy class interactions (e.g. pedestrian vs bus) receive a
10 % RI boost before final classification.

### Single-object events

| Event | Trigger |
|---|---|
| Hard Braking | Speed ≥ threshold AND deceleration ≤ `hard_brake_mps2` |
| Sudden Swerve | Average lateral acceleration + heading change sustained over `swerve_eval_frames`, confirmed for `swerve_persist_frames` consecutive frames |

All parameters live in `config.py` as a single `AppConfig` dataclass.
Key groups:

| Group | Key parameters |
|---|---|

| Border guard | `frame_border_margin` |
| Motion | `velocity_alpha`, `accel_alpha`, `max_speed_mps`, `min_speed_for_risk_mps` |
| Proximity gates | `proximity_gate_m`, `min_closing_speed_mps`, `cpa_miss_distance_m` |
| Lateral gates | `lat_limit_rear_end/merging/crossing/head_on` |
| TTC gates | `ttc_threshold_by_pair` (per class pair) |
| RI scoring | `ri_alpha/beta/gamma`, `ri_distance_ref`, `ri_ttc_ref`, `ri_v_max` |
| Risk thresholds | `ri_high_threshold`, `ri_medium_threshold` |
| Persistence | `incident_persist_frames`, `incident_cooldown_frames` |
| Swerve | `swerve_eval_frames`, `swerve_persist_frames`, `swerve_heading_delta_min_deg` |

## Tuning Guide

### Too many false High/Medium alerts

- Increase `incident_persist_frames` (require more sustained evidence)
- Increase `min_closing_speed_mps` (reject slow-moving adjacent pairs)
- Increase `ri_high_threshold` / `ri_medium_threshold`
- Increase `frame_border_margin` (exclude more of the frame edge)
- Increase `min_track_age_for_risk` (wait longer for velocity to stabilise)
- Reduce `proximity_gate_m` (tighter spatial pre-filter)

### Missing genuine incidents

- Reduce `ri_high_threshold` / `ri_medium_threshold`
- Reduce `incident_persist_frames`
- Increase `proximity_gate_m`
- Reduce `model_conf` (more detections)
- Verify BEV calibration quality

### Hard braking too noisy

- Make `hard_brake_mps2` more negative (e.g. `-4.5` → `-5.5`)
- Increase `hard_brake_min_speed_mps`

### Swerve false positives

- Increase `swerve_eval_frames` (longer confirmation window)
- Increase `swerve_heading_delta_min_deg` (require a larger turn)
- Increase `swerve_lateral_acc_threshold`

### Unrealistic distances / TTC values

- Re-run `bev_calibrator.py` — bad homography is the most common cause
- Verify the scale calibration distance input was correct
- Check that the same camera / video is being used as during calibration

## Common Issues

### Missed detections at distance

The default COCO-pretrained YOLO model loses sensitivity for small/distant
objects. Consider fine-tuning on traffic-specific data or switching to a
model with an additional small-object detection head.

### Frequent track ID switches

- Tune `match_thresh` and `track_buffer`
- ID switches propagate into velocity estimates — increasing
  `min_track_age_for_risk` reduces downstream false alerts from new IDs

### BEV positions look wrong

- The bottom-centre of the bounding box is used as the ground contact point
  and projected through the homography. If your camera angle is very oblique,
  re-calibrate with points closer to the region of interest.

## Future Scope

### Detection & Tracking
- Fine-tune detection model for consistent small/distant object detection
- Add re-identification embeddings to reduce ID switches after occlusion
- Evaluate BoT-SORT as a drop-in tracker replacement for improved ID stability

### Motion & Forecasting
- Replace 1-step TTC with multi-step trajectory forecasting (Kalman CA model)
- Add curvature and lane-aware trajectory constraints for intersections

### Scene Context
- Lane / road-surface segmentation for context-aware lateral gating
- Camera-pose estimation to improve BEV accuracy without manual calibration

### False Positive Reduction
- Adaptive thresholds by traffic density and time-of-day
- Lightweight ML severity model trained on TTC trend, distance trend,
  heading conflict, speed delta, braking profile

### Productisation
- RTSP live stream input with real-time alert output
- REST API for incident query and analytics
- Dockerised deployment with CPU / GPU profiles
